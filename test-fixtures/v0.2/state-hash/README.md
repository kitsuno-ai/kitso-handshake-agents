# state_hash test vectors (v0.2.1)

Golden test vectors for the `state_hash` primitive specified in the Kitso
Handshake protocol §2 (see the [kitso-handshake](https://github.com/kitsuno-ai/kitso-handshake)
spec repo) and implemented in
[`packages/kitso-state-hash`](../../../packages/kitso-state-hash).

Every implementation in every language **must** reproduce these vectors
byte-for-byte. If your hash differs, your canonicalization or JCS encoding
diverges from the spec.

## File format

Each vector file is a JSON object with:

| field | type | meaning |
|---|---|---|
| `kind` | `"vacancy"` \| `"seeker"` | which canonical-subset rule applies |
| `notes` | string | what this vector exercises |
| `input` | object | the full input card (or partial card) before canonicalisation |
| `canonical` | object | the canonical-subset projection of `input` |
| `canonical_bytes_utf8` | string | the JCS-encoded `canonical` as a UTF-8 string (for byte-level inspection) |
| `state_hash` | string | lowercase hex SHA-256 of `canonical_bytes_utf8` as UTF-8 bytes |

## Verifying an implementation

For each vector:

1. Compute `canonical_subset_<kind>(input)` and assert it equals `canonical`.
2. Compute `JCS(canonical)` and assert the UTF-8 representation equals
   `canonical_bytes_utf8`.
3. Compute `lowercase_hex(SHA256(JCS(canonical)))` and assert it equals
   `state_hash`.

`equivalences.json` lists groups of vectors that **must** produce the same
`state_hash` despite different inputs (cosmetic noise, list-order changes,
split-vs-flat language shape, etc).

`index.json` lists all vector files for programmatic discovery.

## Adding new vectors

Vectors are generated by `packages/kitso-state-hash/scripts/build_test_vectors.py`
(soon — currently this set was bootstrapped from the v0.2 fixtures and
hand-crafted edge cases). Once that script lands, regenerate after any rule
change and commit the diff.
